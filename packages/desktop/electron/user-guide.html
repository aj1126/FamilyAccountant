User Documentation & Guides

A premium interactive manual explaining household finance management, custom currencies, and offline-first synchronization.

1. Welcome & Quick Start

FamilyAccountant is a robust, premium personal finance solution designed for families and shared households. Key actions to start recording data include:

  1. Register/Login: Launch the client and authenticate. Session checking validates credentials natively on startup.
  2. Household Selection: Join an existing household using a Join ID UUID shared by a family member, or create a new one to begin inviting household members.
  3. Add Accounts: Create accounts representing actual cash drawers, checking accounts, or credit cards. Specify their default currency (e.g. USD, EUR).
  4. Record Transactions: Record transactions with category descriptions. Accounts auto-adjust balances in response.

2. Household Management

To enforce strict data confidentiality, all transactions, accounts, debts, and payments are scoped exclusively to a Household.

  • Strict Gating: The backend uses a global HouseholdMemberGuard. All operations check that the client's user has a non-null household ID and restrict cross-household reads/writes.
  • Inviting Members: Provide other members your household's Join ID. Once joined, everyone shares real-time access to transaction histories, debt settlements, and accounts.

3. Accounts & Custom Currencies

Configure checking, savings, cash, or credit accounts to match your physical assets. We support strict multi-currency compliance:

  • Currency Scoping: Create accounts with separate ISO codes (e.g. USD, EUR). New transactions created under an account inherit its currency default automatically.
  • Multi-Currency Dashboard: The Dashboard spent statistics group totals by currency. FamilyAccountant avoids arbitrary conversions, ensuring distinct currencies are summed separately to preserve ledger accuracy.

4. Transactions & Balance Control

Transactions added under checking or cash accounts adjust account balances dynamically. The system hooks database level events to automate adjustments:

  • Additions: Recording an expense subtracts amount; recording an income adds it.
  • Edits/Deletions: Changing transaction amounts or moving them between accounts automatically calculates deltas and updates the affected balances.
  • Sync Status Badges:
Synced

Transaction is successfully saved on cloud database.

Pending

Saved in local SQLite cache, waiting to push to backend server.

Error

Synchronization failed. The app will keep records cached locally and attempt a retry on reconnect.

5. Debts & Payments

Record mutual IOUs, track balances, and settle debts within the household.

  • Counterparty Validation: The debtor and creditor in a debt record must be two distinct users within the same household.
  • Payment Tracking: Payment entries against a debt record must not exceed the remaining unpaid balance.
  • Auto-Settling: When cumulative payments equal or exceed the debt principal, the debt state automatically toggles to Settled.

6. Offline Synchronization Architecture

FamilyAccountant implements an offline-first sync mechanism ensuring high availability even when disconnected from the cloud.

  • Instant App Hydration: On launch, the client hydrates its state directly from the local SQLite database. The UI is fully interactive immediately without waiting for API loads.
  • Reconnect Triggers: Network connection change events trigger sync automatically upon returning online.
  • Metadata Control: The client stores lastSyncedAt. Succeeding requests query items modified after this timestamp to minimize payload sizes.

7. Frequently Asked Questions

What happens if my sync fails due to network conflicts?

The transaction is preserved locally with an "Error" state badge. The synchronization queue will run a retry on subsequent actions or when the network connection becomes internet-reachable.

How does the app know which transactions to pull?

The application sends a lastSyncedAt timestamp. The server returns only transaction records created or updated after that time, which the client merges into its SQLite database.

Can household members see other members' passwords?

No. Passwords are salted and hashed on the backend using bcryptjs. The /households/members endpoint strips all password hashes before returning user lists, ensuring strict user credential isolation.
